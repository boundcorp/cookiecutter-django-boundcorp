# Migrating Existing Django Apps into a Cookiecutter Project

Guide for porting Django apps from a standalone project into a project generated by this cookiecutter template.

## Overview

The cookiecutter generates a project with specific patterns (model mixins, admin decorators, JWT auth, pgserver, etc.). Existing Django apps need to be adapted to these patterns when migrating.

## Step-by-Step

### 1. Generate the new project

```bash
cd /tmp
cookiecutter /path/to/cookiecutter-django-boundcorp --no-input \
  project_name=myproject \
  docker_image_url=ghcr.io/org/myproject \
  production_hostname=myproject.example.com \
  author="Your Name" \
  email="you@example.com"
```

### 2. Set up the new project in your repo

```bash
# Back up your database first!
cp db.sqlite3 db.sqlite3.backup

# Copy generated project into repo root
cp -r /tmp/myproject/* .
cp /tmp/myproject/.* . 2>/dev/null

# Install and verify baseline works
make venv
python manage.py migrate
make test
```

### 3. Port each app

Move app code into `myproject/apps/<app_name>/`. Apply these transformations:

#### Import paths

```python
# Old
from clients.models import Client
from invoicing.models import Invoice

# New
from myproject.apps.clients.models import Client
from myproject.apps.invoicing.models import Invoice
```

#### Admin registration

```python
# Old
from django.contrib import admin

@admin.register(Client)
class ClientAdmin(admin.ModelAdmin):
    ...

# New
from django.contrib import admin
from myproject.utils.admin import register

@register(Client)
class ClientAdmin(admin.ModelAdmin):
    ...
```

The `@register` decorator auto-sets `raw_id_fields` for all relation fields.

#### Model mixins

```python
# Old — manual timestamp fields
class Client(models.Model):
    name = models.CharField(max_length=255)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

# New — use TimestampMixin (provides created_at, updated_at, ordering)
from myproject.utils.models import TimestampMixin

class Client(TimestampMixin, models.Model):
    name = models.CharField(max_length=255)
```

Available mixins from `utils/models.py`:
- `TimestampMixin` — `created_at`, `updated_at` with default ordering
- `ShortIdMixin` / `MediumIDMixin` / `UUIDMixin` — string/UUID primary keys
- `ArchivedMixin` — soft delete via `archived_at`
- `ModelDiffMixin` — track field changes

#### User model references

The cookiecutter uses a custom User model with string PKs (MediumIDMixin). Update all User foreign keys:

```python
# Old
from django.contrib.auth.models import User

class TeamMember(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE)

class Invoice(models.Model):
    created_by = models.ForeignKey(User, on_delete=models.SET_NULL, null=True)

# New
from django.conf import settings

class TeamMember(models.Model):
    user = models.OneToOneField(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)

class Invoice(models.Model):
    created_by = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.SET_NULL, null=True)
```

In test code, import the User model directly:
```python
from myproject.apps.users.models import User
```

#### FK string references

String references in ForeignKey fields use `app_label`, which doesn't change:
```python
# These stay the same — Django resolves by app_label, not full path
client = models.ForeignKey("clients.Client", on_delete=models.CASCADE)
```

### 4. Register apps in settings

Edit `myproject/settings/project.py`, add to `INSTALLED_APPS`:

```python
INSTALLED_APPS = [
    ...
    "myproject.apps.clients",
    "myproject.apps.projects",
    "myproject.apps.invoicing",
]
```

### 5. Mount API routers

Edit `myproject/api/__init__.py`:

```python
from myproject.apps.clients.api import router as clients_router
api.add_router("/clients", clients_router)
```

### 6. Make migrations

```bash
python manage.py makemigrations clients projects invoicing
python manage.py migrate
```

### 7. Migrate data

If you're migrating from SQLite to pgserver (or external Postgres), you can't use `loaddata` directly because the User model PKs changed from integers to strings.

**Option A: Django dumpdata/loaddata** (works if User PKs are compatible)
```bash
# From old project
python manage.py dumpdata --indent 2 --natural-foreign app1 app2 > fixtures/data.json
# In new project
python manage.py loaddata fixtures/data.json
```

**Option B: Custom migration script** (needed when User PKs changed)

Write a management command or shell script that:
1. Connects to the old database (sqlite3 module)
2. Creates new User records (gets new string PKs)
3. Builds a mapping: `old_user_int_pk → new_user_string_pk`
4. Iterates through each table, creating records with remapped FKs

```python
import sqlite3
conn = sqlite3.connect('db.sqlite3.backup')
conn.row_factory = sqlite3.Row

# Create users in new DB, build mapping
old_to_new_user = {}
for row in conn.execute('SELECT * FROM auth_user'):
    new_user = User.objects.create_user(row['username'], row['email'], 'temp')
    old_to_new_user[row['id']] = new_user

# Import other models, remapping user FKs
for row in conn.execute('SELECT * FROM myapp_model'):
    MyModel.objects.create(
        ...,
        created_by=old_to_new_user.get(row['created_by_id']),
    )
```

## Patterns to Follow

- **Admin URL**: `/mgmt/` (not `/admin/`)
- **API URL**: `/api/` with interactive docs at `/api/docs`
- **Auth**: JWT via `ninja_jwt` (not session auth)
- **Tests**: pytest + factory-boy, `@pytest.mark.django_db` on test classes
- **PDF generation**: Use `utils/services/render_pdf.py` (xhtml2pdf)
- **Settings**: Adaptive — no env vars = pgserver + filesystem + eager celery

## Testing the Migration

```bash
# Local tests (pgserver)
make test

# Docker integration tests (real postgres, redis, garage)
make test_aio
```
